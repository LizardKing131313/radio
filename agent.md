# Говновоз Радио - контекст и правила для Codex агента

## Назначение документа

Этот документ - источник истины для агента, который работает над проектом. Его задача - не рассуждать абстрактно, а
принимать решения в рамках уже выбранного направления и выдавать практичный результат.

Главный критерий качества здесь не количество слоев, паттернов и файлов, а то, насколько быстро и надежно получается
рабочая система.

## Главная цель проекта

Нужно собрать автономное 24/7 радио с AI-каверами, которое:

- заранее находит и качает треки с YouTube
- держит аудио в кешах
- проигрывает поток непрерывно через Liquidsoap
- mux-ит поток в HLS через FFmpeg
- отдает HLS на сайт
- при необходимости параллельно пушит YouTube Live
- умеет показывать текущий трек
- в будущем умеет обновлять описание live и чат на YouTube

Проект должен быть пригоден для реального продакшена, а не для демки на коленке.

## Главное правило проекта

**Чем меньше кода - тем лучше.**

Это не шутка и не пожелание. Это главный инженерный фильтр для любого решения.

Агент обязан выбирать решение, которое:

- делает задачу минимальным количеством кода
- проще поддерживается
- проще читается
- проще дебажится
- требует меньше скрытой магии
- не тащит лишние зависимости
- не плодит абстракции без прямой пользы

Если две реализации одинаково рабочие, выбирать надо ту, где:

- меньше файлов
- меньше состояний
- меньше сущностей
- меньше конфигов
- меньше точек отказа

Нельзя раздувать проект ради красоты архитектуры. Нужен рабочий мотор, а не выставка паттернов.

## Технологическая цель проекта

Цель - взрослый deployable стек, а не набор локальных скриптов.

Актуальный целевой стек:

- Python 3.11+
- Kubernetes как orchestration слой
- Liquidsoap
- FFmpeg
- yt-dlp
- PostgreSQL там, где действительно нужна нормальная постоянная модель данных
- Alembic как единственный владелец схемы БД
- FastAPI как тонкий HTTP-слой
- YouTube Data API для поиска и простой информации о треках
- Redis только если появится реальная задача на volatile state, lock, cache, coordination или short-lived data
- Nginx как раздача HLS и reverse proxy

Важно:
Docker - это упаковка, Kubernetes - владелец жизненного цикла процессов. Python не должен снова становиться
самописным Kubernetes.

## Что считать зрелым вариантом, а что нет

Зрелый вариант:

- один kustomize deployment root
- явные Kubernetes workloads
- явные PVC/emptyDir volumes
- нормальные readiness checks
- статический config через ConfigMap
- секреты через Kubernetes Secret
- реальные секреты только в ignored `deploy/k8s/secret.yaml`, в репе только `secret.example.yaml`
- предсказуемые пути данных
- приложение можно поднять одной командой
- данные и кеши не теряются случайно

Незрелый вариант:

- shell-скрипты как основной orchestrator
- хранение важного состояния в txt-файлах без нужды
- SQLite просто потому что так быстрее написать, если уже понятно, что будет queue, offers, moderation и API
- хаос из папок, которые живут только в голове автора

## Правило выбора хранилищ

Агент не должен пихать PostgreSQL и Redis везде подряд. Надо использовать их только там, где они реально нужны.

### PostgreSQL использовать для:

- каталога треков
- очереди треков
- предложек от пользователей
- модерации офферов
- истории воспроизведения, если она нужна
- административных и доменных данных, где важны целостность, индексы, фильтры и нормальные запросы

### Redis использовать для:

- кэша короткоживущих данных
- distributed lock, если будет несколько воркеров
- коротких флагов состояния
- rate limit
- ephemeral now playing cache, если это действительно упрощает задачу
- координации между процессами, если без этого уже никак

### Не использовать PostgreSQL и Redis для:

- всего подряд просто ради взрослого вида
- хранения файлов
- замены обычной файловой системы там, где нужен просто cache dir
- усложнения MVP без реальной пользы

Best practice для этого проекта:

- файлы аудио и HLS сегменты живут в файловой системе
- доменные сущности живут в PostgreSQL
- короткоживущее состояние и технический кэш при необходимости живут в Redis

## Текущее направление разработки

Текущее направление - убрать самописную оркестрацию из Python и держать runtime в Kubernetes.

Python делает только прикладную работу:

- поиск через YouTube Data API
- prefetch/download через yt-dlp
- репозитории PostgreSQL
- сбор аргументов FFmpeg

Запуском и перезапуском процессов занимаются Kubernetes/kubelet.

## Базовый пайплайн

```text
YouTube
  -> YouTube Data API search
  -> PostgreSQL tracks
  -> yt-dlp prefetch
  -> cold cache
  -> hot cache
  -> Liquidsoap
  -> FIFO / pipe
  -> FFmpeg
  -> HLS TS + CMAF
  -> Nginx
  -> сайт

PostgreSQL -> FastAPI -> Nginx /api/ -> админка/клиент
```

Опционально:

```text
FFmpeg -> RTMP -> YouTube Live
```

## Архитектурный стиль

Используем гексагональный подход, но без фанатизма.

Практический смысл такой:

- доменная логика отдельно
- инфраструктурные адаптеры отдельно
- API отдельно
- процесс-менеджмент отдельно

Но нельзя ради этого устраивать фабрики фабрик и интерфейсы интерфейсов.

Нормально:

- service layer
- adapters
- repositories
- config

Ненормально:

- шесть абстрактных базовых классов на один subprocess
- generic repository ради двух запросов
- event bus ради локального менеджера радио

## Целевая инфраструктура

Проект должен быть рассчитан на запуск в Kubernetes через `kubectl apply -k deploy`.

Минимальный контур:

- Postgres StatefulSet - основная БД
- Alembic Job - миграции схемы
- radio Deployment - один pod с контейнерами search, prefetch, api, liquidsoap, ffmpeg, nginx
- PVC - durable audio cache
- emptyDir - FIFO/runtime/HLS output

Не добавлять Redis, operators, service mesh, message broker и отдельные контроллеры без конкретной пользы.

Best practice сейчас: один tightly coupled `radio` pod с несколькими контейнерами и общими volumes. Это меньше кода,
чем Python runner, и меньше инфраструктуры, чем разносить каждый процесс по отдельным сервисам.

## Требования к коду

### Общие требования

- Python 3.11+
- UTF-8 без BOM
- строгие type hints везде
- PEP8
- читаемые имена переменных
- никаких однобуквенных имен
- никаких имен сущностей, файлов и переменных, завязанных на тематику канала
- runtime output и пользовательские строки в коде - только на английском
- комментарии можно на русском

### Обязательные правила проектирования кода

- чем меньше кода, тем лучше
- чем меньше сущностей, тем лучше
- чем меньше магии, тем лучше
- чем меньше неявного поведения, тем лучше
- чем меньше дублирования, тем лучше
- чем меньше специальных кейсов, тем лучше

### Что агент обязан делать при генерации кода

- сначала выбирать самое простое рабочее решение
- не писать код на будущее, если это будущее не подтверждено задачей
- не создавать абстракции раньше времени
- не дробить код на десятки файлов без реальной причины
- не использовать паттерны только потому, что они красивые
- не плодить конфиги, если хватает одного settings слоя
- не тащить библиотеки, если задача решается стандартной библиотекой

### Что считается хорошим кодом для этого проекта

Хороший код здесь это код, который:

- можно прочитать через месяц без боли
- можно быстро починить ночью
- можно логически проследить по шагам
- не требует держать в голове десять скрытых зависимостей
- не заставляет искать бизнес-логику по всему проекту

### Что считается плохим кодом для этого проекта

Плохой код здесь это код, который:

- слишком умный для своей задачи
- завязан на неявные side effects
- размазывает логику по util-мусорке
- превращает запуск процесса в квест
- требует рефакторинга еще до первого деплоя

## Формат кода, который должен выдавать агент

Если пользователь просит код:

- выдавать полный файл
- без diff-мусора
- без многоточий и обрезков
- с типами
- с импортами
- в готовом к вставке виде

Если файлов несколько:

- выдавать списком имя файла -> содержимое

Если задача спорная:

- кратко назвать лучший вариант
- объяснить в 2-4 фразах почему он лучший
- дальше сразу давать решение

## Где и как использовать БД

На текущем взрослом контуре правильнее ориентироваться на PostgreSQL как основное постоянное хранилище.

Что должно жить в PostgreSQL с прицелом на нормальный прод:

- tracks
- track_sources
- queue_entries
- offers
- playback_events
- maybe channel_settings / stream_state metadata

Даже если часть этого сначала не реализована, агент должен думать в сторону нормальной схемы, а не времянки, которую
потом больно выкидывать.

Но это не значит, что надо сразу моделировать полмира. Нужна компактная схема без лишних таблиц.

## Где и как использовать Redis

Redis в этом проекте вторичен. Он не должен быть источником истины для доменной модели.

Правильная роль Redis здесь:

- cache
- lock
- transient state
- debouncing / throttling
- fast technical coordination

Неправильная роль Redis здесь:

- хранить единственную очередь треков без постоянной БД
- быть единственным местом, где живет критичное состояние
- заменять нормальную модель данных

## Где и как использовать файловую систему

Файловая система здесь полноценная часть системы.

В файловой системе должны жить:

- cold cache аудио
- hot cache аудио
- FIFO или unix pipe артефакты
- HLS output
- runtime info files вроде nowplaying.txt, если это самый простой и надежный способ интеграции

Не надо пытаться утащить это в БД или Redis. Это будет тупо и больно.

## Процессы внутри приложения

Приложение больше не является управляющим сервисом. Управляющий слой - Kubernetes.

Долгоживущие процессы:

- `python -m manager search`
- `python -m manager prefetch`
- `uvicorn manager.api:app --host 0.0.0.0 --port 8000`
- `liquidsoap -v /opt/radio/data/radio.liq`
- `python -m manager ffmpeg-hls`, который делает `execvp(ffmpeg)`
- `nginx`

В Python не должно быть `Runner`, `ControlBus`, `ServiceRunnable`, subprocess supervisor или локального API управления
процессами без новой явной причины.

## Требования к процесс-менеджменту

- запуск и restart policy описывать в Kubernetes
- health/readiness делать простыми и проверяемыми
- Python worker должен падать при фатальной ошибке, а не скрывать проблему бесконечной внутренней магией
- FFmpeg/Liquidsoap запускать как готовые процессы, без лишних Python-оберток

## Требования к API

Минимальный API:

- `GET /current`
- `GET /queue`
- `POST /queue/append`
- `POST /queue/append/admin`
- `POST /offers/add`
- `GET /offers`
- `GET /offers/{id}`
- `POST /offers/{id}/accept`
- `POST /offers/{id}/cancel`
- `GET /health`

API должно быть тонким слоем. Вся логика должна жить не в роутерах.
API не должен управлять процессами, пока это не станет отдельной явной задачей.

## Требования к now playing

nowplaying.txt - нормальная интеграционная точка, если это самый короткий путь.

Но агент должен помнить, что есть разница между:

- тем, что уже стартовало в Liquidsoap
- тем, что дошло до сайта через HLS лаг
- тем, что слышно в YouTube Live

Значит `/current` должен учитывать offset, а не тупо читать последний текстовый файл и молиться.
Текущая реализация читает Liquidsoap `nowplaying.txt(.kv)` и отдает расчетный HLS offset.

## Приоритеты разработки

### Делать сейчас

- держать Kubernetes manifests минимальными
- держать `manager/main.py` маленьким CLI для контейнерных команд
- держать FastAPI тонким слоем над репозиториями
- держать PostgreSQL schema в Alembic
- держать поиск на YouTube Data API
- держать YouTube API quota/error telemetry видимой через `/health`
- держать скачивание на yt-dlp только в prefetch
- документировать реальный runtime в `docs/runtime.md`

### В беклог

- search index
- metadata sync для YouTube live
- retention policies

### Перфекционизм

- metrics
- tracing
- web admin
- richer observability
- autoscaling fantasies, которые пока нахуй не нужны

### Похуй

- premature microservices
- overengineered DDD circus
- отдельный message broker без реальной причины
- сложные generic abstractions

## Что считать успехом

Успех - это когда проект:

- поднимается через `kubectl apply -k deploy`
- стабильно играет
- переживает падение процессов
- не разваливается от перезапуска
- хранит важные данные в нормальном месте
- не превращается в свалку магии
- можно расширять без переписывания ядра

## Короткая формула для агента

Этот проект - не про красивую теорию.

Это Kubernetes-deployable 24/7 радио: Python отвечает за поиск, prefetch и БД, а Kubernetes запускает Liquidsoap,
FFmpeg, Nginx и workers. PostgreSQL используется для доменных данных, Redis добавляется только под реальную задачу.
Главный критерий - минимальное по коду и конфигам рабочее решение.

Если агент уходит в сторону лишней сложности, он ошибся.
